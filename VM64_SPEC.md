# VM64 Specification Notes

## ABI Function Values

ABI `function` values are unsupported in VM64.

In the Ethereum ABI, an external function value is encoded as:

```text
address20 || selector4
```

This fits in one 32-byte ABI word. In VM64, QRL contract addresses are 64
bytes, so the same value would be:

```text
address64 || selector4
```

That is 68 bytes, which does not fit in one 64-byte ABI slot. VM64 does not
define a custom two-slot or dynamic encoding for this type.

Implementation policy:

- ABI JSON containing `type: "function"` may still parse.
- Packing ABI `function` values fails.
- Unpacking ABI `function` values fails.
- Reconstructing indexed event topics for ABI `function` values fails.
- Go binding generation fails when a contract ABI contains an ABI `function`
  value in constructor inputs, method inputs or outputs, event inputs, custom
  error inputs, arrays, or tuples.

Normal Solidity function declarations are unaffected. Selectors, dispatch, and
ordinary calls such as `function transfer(address to, uint256 amount)` keep
their normal meaning.

Internal Solidity function pointers are also unaffected by this ABI policy.
They are compiler-local bytecode implementation details and are not encoded as
external ABI values.

## ABI Fixed Bytes

VM64 supports fixed bytes from `bytes1` through `bytes64`.

Fixed bytes are static ABI values. Each value occupies one 64-byte ABI slot,
is copied from the left, and is right-padded with zeros when the declared size
is smaller than 64 bytes. `bytes64` therefore fills the entire slot and maps to
Go `[64]byte` in ABI helpers and generated bindings.

`bytes64` can carry raw address-sized blobs because QRL addresses are 64 bytes,
but `address` remains the semantic ABI type for account and contract addresses.

## Address Format and IBAN / ICAP

The canonical VM64 account and contract address format is a Q-prefixed 64-byte
QRL address:

```text
Q || 128 lowercase or uppercase hex characters
```

Ethereum-style IBAN/ICAP address encoding is unsupported in VM64. Direct ICAP
encoding cannot represent a 64-byte QRL address within the legacy address field,
and indirect ICAP name-registry transfer flows are not part of Hyperion VM64.

Tooling must not silently convert IBAN/ICAP strings into addresses. Legacy
console APIs such as `web3.isIBAN`, `web3.fromICAP`, `web3.qrl.iban`,
`web3.qrl.sendIBANTransaction`, and `web3.qrl.icapNamereg` are removed or must
fail clearly if reached through an obsolete bundled dependency.

## ABI Integers

ABI integer type names are real declared widths, not aliases for the VM word.

Unsigned integer types `uintN` and signed integer types `intN` are accepted
only for byte-aligned widths from 8 through 512 bits. `uint512` and `int512`
are the full 64-byte VM word integer types.

All integer ABI values still occupy one 64-byte ABI slot. Narrower unsigned
types must be zero-extended to that slot, and narrower signed types must be
sign-extended to that slot.

Encoding rejects values outside the declared type range. For example,
`uint256` rejects values above `2^256 - 1` even though they could fit in a
64-byte ABI slot. Decoding also rejects non-canonical slot values whose high
bits are not valid zero/sign extension for the declared type.

## Built-In Panic Reverts

Built-in panic reverts use the Solidity-reserved signature `Panic(uint256)`.

The selector remains:

```text
keccak256("Panic(uint256)")[:4]
```

The panic code is encoded using the Hyperion ABI slot width: one 64-byte ABI
word. The code must fit the declared `uint256` range; high bits above 256 are
invalid. The canonical panic revert payload length is therefore:

```text
4-byte selector + 64-byte argument = 68 bytes
```

`Panic(uint512)` is not a built-in panic signature. `uint512` remains available
for ordinary ABI values, but built-in panic codes are small compiler/runtime
diagnostic constants and keep the `uint256` signature.

## Hyperion Solidity Codegen

Hyperion Solidity and related code generators must emit VM64-native bytecode and
ABI metadata. The node does not provide an Ethereum-compatibility mode for
compiler output.

Compiler and codegen output must match these node-side rules:

- ABI encoder and decoder logic use 64-byte ABI slots.
- Built-in panic reverts use `Panic(uint256)` with a 68-byte payload:
  4-byte selector plus one 64-byte ABI argument.
- `address` values are 64-byte QRL addresses.
- Fixed bytes types support `bytes1` through `bytes64`; `bytes64` occupies one
  full ABI slot.
- Integer types support `uint512` and `int512` as full VM-word integers, while
  narrower declared widths such as `uint256` keep their narrower ranges.
- CREATE2 uses a 64-byte salt and the QRL address derivation rules defined by
  the node, including the domain-separated Keccak512 contract address scheme.

Generated bytecode fixtures compiled for Ethereum's 32-byte ABI slots, 20-byte
addresses, 32-byte panic payloads, or Ethereum CREATE2 address derivation are
legacy artifacts and must not be treated as canonical VM64 compiler output.
Binding tests that only need deploy-wrapper type generation may use explicit
compile-only placeholder bytecode, but executable bytecode fixtures must be
regenerated from Hyperion Solidity/compiler output.

## Gas Word Pricing

VM64 uses the VM word size for word-priced execution gas.

The canonical VM gas word is `uint512.WordBytes`, currently 64 bytes. All code
paths that round byte lengths up to "words" for execution gas must use that
width. This includes memory expansion, copy opcodes, initcode word charging,
`KECCAK256`/`CREATE2` word charging, and precompile input word charging where the
precompile uses the shared VM `toWordSize` helper.

The existing gas constants are intentionally not doubled. For example,
`MemoryGas`, `CopyGas`, `Keccak256WordGas`, `InitCodeWordGas`,
`Sha256PerWordGas`, and `IdentityPerWordGas` keep their numeric values while the
unit they multiply is now a 64-byte VM word. This deliberately lowers the
effective per-byte price of word-priced operations compared with a 32-byte-word
VM and also changes the memory quadratic curve because fewer words are counted
for the same byte length.

This policy does not change gas charged directly per byte. Transaction calldata
zero/non-zero byte gas and LOG data gas remain byte-priced.

## Economic Quantities

Economic quantities remain 256-bit.

VM64 widens VM stack words, ABI slots, addresses, storage values, and log
topics. It does not widen economic quantity domains such as account balances,
transaction values, gas prices, fee caps, base fees, rewards, or block number
quantity encodings.

RPC quantity fields for these values may continue using `hexutil.Big` and must
reject values above 256 bits.

## Execution Golden Coverage

VM64 requires an independent execution golden-test suite.

Targeted tests cover individual migration points, but they are not a complete
VM64 conformance suite. The broad golden coverage must include:

- arithmetic edge cases at `2^512`, including overflow, underflow, wrapping, and
  zero-boundary behavior;
- `SIGNEXTEND`, `SAR`, and signed comparison opcodes across 512-bit positive,
  negative, minimum, and maximum values;
- memory expansion gas under 64-byte word sizing, including boundary crossings
  and quadratic-growth cases;
- `SSTORE` and `SLOAD` round-trips for full 64-byte storage values while storage
  keys remain 32 bytes;
- log emission and filtering with 64-byte topics;
- ABI encoding, decoding, and topic handling for `uint512` and `int512`;
- revert payloads, including `Error(string)` and canonical `Panic(uint256)`
  payloads with one 64-byte argument slot.

Some of this coverage exists in targeted unit tests. The remaining task is to
consolidate it into a complete VM64 execution conformance suite with independent
golden vectors.

## Storage Keys And Values

VM64 storage slot keys remain 32 bytes.

`SLOAD` and `SSTORE` receive a 64-byte stack word as the slot operand, but the
storage key is the low 32 bytes of that word:

```text
storageKey32 = low256(stackWord512)
```

The high 32 bytes are ignored for storage-key selection. Two slot operands that
differ only in the high 32 bytes intentionally address the same storage slot.

Storage values are 64 bytes. A full 512-bit VM word, including a 64-byte
address, round-trips through `SSTORE` and `SLOAD` without truncation.

Host-facing APIs use the same split:

- access-list storage keys are 32-byte `common.Hash` values;
- genesis and state override storage maps are keyed by 32-byte `common.Hash`;
- proof requests identify storage keys as 32-byte hashes;
- proof and state values are encoded as fixed 64-byte storage values.

Keeping keys at 32 bytes preserves the Keccak256-derived storage-key domain used
by compiler-generated mappings, arrays, structs, and nested slot paths.

## Hyperion Typed Data

Hyperion typed data is derived from EIP-712, but its field encoding is
VM64-native.

The JSON envelope keeps the familiar `types`, `primaryType`, `domain`, and
`message` fields. The domain type name remains `EIP712Domain` for compatibility
with existing typed-data JSON producers, but the encoded preimage is not the
Ethereum 32-byte-word layout.

Encoding policy:

- Every encoded typed-data member occupies one 64-byte ABI word.
- `address` values are 64-byte QRL addresses and fill one 64-byte word.
- `bool` and integer values are left-padded/extended into one 64-byte word.
- Fixed bytes values support `bytes1` through `bytes64` and are right-padded
  into one 64-byte word.
- Dynamic `bytes`, `string`, arrays, nested structs, and the type hash still
  use Keccak256 hash values, but each 32-byte hash is encoded as a fixed
  `bytes32` value in a 64-byte word before the containing struct is hashed.
- Integer type names are real declared widths from 8 through 512 bits.
  `uint512` and `int512` are the full VM64 integer types. Narrower integer
  names, such as `uint256`, keep their narrower ranges and are not aliases for
  a 64-byte word.
- Bare typed-data `uint` and `int` are treated as VM64 word aliases:
  `uint512` and `int512`.

The final typed-data digest still follows the EIP-191 style envelope:

```text
keccak256(0x1901 || domainSeparator32 || messageHash32)
```

where `domainSeparator32` and `messageHash32` are Keccak256 hashes of the
VM64-native encoded domain and message structs.

## RPC Log Topics

RPC log filters are VM64-native.

Filter criteria for `qrl_getLogs`, `qrl_newFilter`, and log subscriptions accept
only full 64-byte topic hex strings. A 32-byte topic string is malformed input
and must fail instead of being padded or treated as an Ethereum-compatible
topic.

Returned logs also serialize each topic as a 64-byte hex string.
